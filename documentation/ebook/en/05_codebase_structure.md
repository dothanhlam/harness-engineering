# Chapter 5: Codebase Structure & Modules

To effectively work with the Harness Engineering repository, it's important to understand how the codebase is organized. The repository is divided into structural components that power the AI and functional packages that the AI generates.

## The Core Files

### `main.go`
This is the heart of the orchestrator. It manages the state machine, delegates tasks to agents, enforces security guardrails (`AuditGeneratedCode`), and captures telemetry.

For example, the pipeline state is strictly tracked using this `WorkflowState` struct, which gets dumped to `state.json` at every stage transition:
```go
// WorkflowState is the persisted pipeline state written to workspace/state.json.
type WorkflowState struct {
	TaskID       string    `json:"task_id"`
	CurrentStage Stage     `json:"current_stage"`
	RetryCount   int       `json:"retry_count"`
	UpdatedAt    time.Time `json:"updated_at"`
}
```

It also includes our stringent security audit logic to prevent the AI from doing anything destructive:
```go
if strings.Contains(code, "rm -rf") {
	auditErr = fmt.Errorf("file %s contains destructive terminal command 'rm -rf'", path)
	return fmt.Errorf("audit failure")
}
```

### `harness_config.json`
The global configuration file. It dictates which agents to use (e.g., `gemini`, `agy`, `ollama`), sets the maximum number of self-healing retries, and defines API endpoints.

## The AI's Environment

* **`.agents/`**: Contains the system prompts and behavioral configurations for our AI agents. For example, `antigravity_dev_prompt.md` instructs the Developer agent on our exact coding standards.
* **`memory/`**: This is the "brain" of the AI. Unlike human engineers, AI agents lose their context between runs. We persist their memory here:
    * `definitions_of_done.md`: The strict technical checklist generated by the BA agent.
    * `mem0-server/`: Local Mem0 vector database backend used for semantic search.
    * `lessons_learned.md`: A living document where the AI records bugs it has fixed so it doesn't repeat them.

## The Generated Workspace

* **`workspace/`**: This is the sandbox where all the magic happens. All AI-generated code is placed here. Currently, our AI has successfully generated and verified several highly secure modules:
    * `password/`: A robust bcrypt hashing library optimized for high-performance zero-allocation techniques.
    * `email_validation/`: A strict unit-tested module for email validation.
    * `landing_page/`: A complete, self-contained glassmorphic marketing site.
    * `random/`: A secure random generation utility.
    * `fibonacci/`: A highly optimized arbitrary-precision Fibonacci sequence generator.
    * `state.json`: The tracking file that persists the current execution stage of the pipeline.
    * `telemetry.json`: The analytics payload generated at the end of every successful execution cycle. Here's what we track:
      ```go
      type Telemetry struct {
          TotalDurationSeconds float64  `json:"total_duration_seconds"`
          StagesExecuted       []string `json:"stages_executed"`
          TotalRetriesUsed     int      `json:"total_retries_used"`
          CodeHealingSuccess   bool     `json:"code_healing_success"`
          LinesOfCodeGenerated int      `json:"lines_of_code_generated"`
          Timestamp            string   `json:"timestamp"`
      }
      ```

By keeping the orchestrator (`main.go`) strictly separated from the generated code (`workspace/`), we ensure that our AI can build modular, testable software without ever breaking the core pipeline itself.
